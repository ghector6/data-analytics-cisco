# 2.0.1 Understanding Big Data and Its Management

> [!abstract] Module Introduction
> Module 2, "Data Collection and Storage," aims to describe what massive datasets ([[Big Data]]) are and how they are managed. In this introductory topic, we will explore both the characteristics that define [[Big Data]] and the fundamental techniques used for its pipeline processing, storage, and governance. This serves as the foundation for understanding why massive datasets require a different approach from traditional databases.

## Characteristics of Big Data (The 3 Vs)

[[Big Data]] is not defined solely by the size of the files, but by the complexity inherent in its nature. The industry (and Cisco) use the **"3 Vs"** model to describe it:

| Characteristic | Description | Technical Implication |
|:---|:---|:---|
| **Volume** | The scale of the data is massive. We move from Gigabytes to Terabytes and Petabytes. | Requires distributed infrastructure (such as [[HDFS]]) and [[NoSQL]] databases. |
| **Velocity** | The speed at which data is generated and must be processed (streaming, [[IoT]], real-time transactions). | Requires real-time ingestion systems (e.g., [[Apache Kafka]]) and batch or stream processing. |
| **Variety** | Data is no longer exclusively structured ([[SQL]] tables). It also includes semi-structured ([[JSON]], [[XML]]) and unstructured data (text, images, videos). | Determines the storage strategy: [[Data Lake]] (raw data) vs. [[Data Warehouse]] (structured data). |

## Big Data Management and Pipelines

Managing [[Big Data]] is not just about storing files; it is about orchestrating a continuous workflow. This workflow is known as a [[Data Pipeline]] and follows the [[ETL]] standard ([[Extract]], [[Transform]], [[Load]]):

| Phase | Description | Common Tools |
|:---|:---|:---|
| **Extract** | Obtain data from original sources ([[APIs]], transactional databases, flat files, [[IoT]] sensors). | [[Apache Nifi]], [[Talend]], [[REST API]]. |
| **Transform** | Clean, normalize, enrich, and structure the data. Business rules are applied here, and outliers or null values are corrected. | [[Apache Spark]], [[Python]] ([[Pandas]]), [[SQL]]. |
| **Load** | Store the final result in a destination system optimized for querying or exploitation. | [[Data Warehouse]] ([[Snowflake]], [[Amazon Redshift]]) or [[Data Lake]] ([[Amazon S3]], [[Azure Data Lake]]). |

### Storage Strategies

- **[[Data Lake]]**: Stores raw, unprocessed data. Ideal for *Variety* (images, logs, JSON).
- **[[Data Warehouse]]**: Stores processed and structured data. Ideal for controlled *Volume* and fast querying ([[Business Intelligence]]).

## Reflection and Application Exercises

1. **Reflection:** Why do you think it is more costly (in time and resources) to correct an error during the *Transform* phase than during the *Extract* phase?

2. **Application:** Imagine that you work for an airline company. You need to store flight data (structured) and weather data (unstructured). Which storage strategy ([[Data Lake]] vs [[Data Warehouse]]) would you use for each one and why?

> [!tip] Key Takeaway
> [[Big Data]] is the convergence of immense volume, high velocity, and chaotic variety. Managing it involves orchestrating pipelines ([[ETL]]) that transform chaos into order, while wisely choosing between [[Data Lake]] (for exploration) and [[Data Warehouse]] (for reporting). This combination (characteristics + management) forms the foundation of modern data engineering.
